## 0. TL;DR — 설계 결론

- **코어 카테고리 ~8–9개가 table-stakes:** 온도·길이·질량(무게)·속도·부피·넓이·시간·디지털 저장용량(+연비).
  Google 인라인 위젯과 프라이버시 툴(onDevTools)이 이 집합으로 수렴. 압력·에너지·전력·각도·요리·의류/신발
  사이즈는 **차별화**(baseline 아님).
- **상호작용:** **버튼 없는 실시간 양방향 변환**(입력값·단위 아무거나 바꾸면 즉시 갱신)이 기본. **멀티-타깃
  출력**(한 값을 그 카테고리의 모든 단위로 동시 표시)이 best-in-class 패턴.
- **수요는 소수 쌍에 집중**(cm↔in, kg↔lbs, °C↔°F, km↔mi, mL↔cups, mph↔kph …). 단, **딥한 SEO 현실은
  "[X] to [Y]" 쌍별 롱테일**이고 리더들(unitconverters.net, Omni 326개 계산기)은 **쌍별 페이지 구조**로 이긴다.
  → **우리 차별점은 제네릭이 아니라 현지화·전통단위·숫자표기** 각도(경쟁사 미개척).
- **로케일 차이는 진짜다:** 미국=임페리얼↔미터법, 전 미터법 시장=요리 컵 차이+신체 단위, **DE=콤마 소수점
  포맷**(출력은 `Intl.NumberFormat`이 공짜, **입력 파싱 "1,5"→1.5는 직접 구현**), **KO/JA=전통단위**(평/돈/근,
  坪/畳/貫/匁/斤). **평=坪=동일 단위(정확히 400/121 m²)** → 로케일 간 공유 팩터.
- **UI:** 2단 카테고리 구조(작은 "자주 쓰는" 세트 전면 + 전체를 그룹으로 접기) + from/to 듀얼 입력 + 실시간
  타이핑 + swap/copy/share + (가벼운) 정밀도 + 모바일 숫자 키패드.
- **환율/FX = 범위 밖**(실시간 시세=백엔드). 수요는 크지만 우리 static·$0 제약과 충돌.

## 1. 기능 세트 — table-stakes vs 차별화

### Table-stakes (필수)
- **카테고리 ~8–9개(교집합):** 온도 · 길이 · 질량/무게 · 속도 · 부피 · 넓이 · 시간 · 디지털 저장용량 · 연비.
  (Google 인라인 위젯이 정확히 이 9개. `[3-0 확정]`)
  - **디지털 저장용량은 SI(1000, kB/MB) + IEC(1024, KiB/MiB) 둘 다** 지원.
- **버튼 없는 실시간 양방향 변환:** 입력값 또는 어느 쪽 단위를 바꿔도 즉시 갱신, 단순 변환에 "Convert" 버튼
  없음. from·to 양쪽 필드 모두 편집 가능. `[확정, 4+ 경쟁사 교차확인]`
- **멀티-타깃 출력:** 입력값을 그 카테고리의 **모든 단위로 동시 표시**. `[medium — best-in-class 패턴]`

### 차별화 (baseline 아님 — 선택)
- 추가 카테고리: 압력·에너지·전력·각도·연료효율·**요리/주방**·**신발/의류 사이즈**. (onDevTools가 의도적으로 생략)
- **전통/지역 단위**(KO 평/돈/근, JA 坪/畳/貫/匁) — **경쟁사가 미개척**(onDevTools는 지역단위 0개). ← **우리의 쐐기.**
- swap · copy · 공유 가능 URL 상태 · 즐겨찾기/최근 기록.

### ⚠️ 과투자 금지 (반증됨)
- **정밀도 다이얼 / 반올림 모드 / 15자리 정밀** 를 "기대 기능"으로 본 claim 3개 전부 **반증**(0-3, 1-2).
  → 합리적 기본 정밀도만, 엔지니어용 정밀 컨트롤에 과투자 X. (넣더라도 가볍게.)

## 2. 수요 — 어떤 변환이 많이 검색되나

- **소수의 임페리얼↔미터법 쌍이 지배**(편집 큐레이션 신호). unitconverters.net "Common Conversions" =
  17개 양방향 쌍: **cm↔in, mm↔in, m↔ft, km↔mi, cm↔ft, kg↔lbs, g↔oz, lb↔oz, °C↔°F, L↔gal, mL↔cups,
  mph↔kph**, acres↔sqft, rad↔deg, hp↔kW, m↔yd. `[3-0 확정]`
- **정량 신호 1개(검색 스니펫):** `celsius to fahrenheit` ≈ **월 33만 검색 · KD 23**. (롱테일 쌍은 KD 낮음.)
- **⚠️ 정확한 순위는 미확정:** Omni·앱스토어 "인기" 리스트로 지배 쌍 순위를 확정하려던 claim들은 **반증**(0-3).
  → **특정 순위는 방향성으로만.** 실제 순위·로케일별 차이는 GSC/키워드 도구로 재확인 필요(미해결 질문).
- **SEO 함의:** "[X] to [Y]" **쌍별 롱테일**이 핵심. 리더들은 **쌍당 1페이지**로 이긴다(Omni 326개, 어떤 사례는
  프로그래매틱 SEO로 2,600페이지). → §5 아키텍처 결정 참조.

| 🇩🇪 DE/DACH | **콤마 소수점 표기**("1,5", 천단위 "."). 신체(키 cm·몸무게 kg), 요리 |

### 3.1 DE 숫자 표기 (구현 크리티컬) `[3-0 확정, 환경서 실측]`
- **출력:** `Intl.NumberFormat('de-DE').format(123456.789)` → `"123.456,789"`, `format(1.5)` → `"1,5"`. **공짜.**
- **입력 파싱:** `Intl.NumberFormat`에 **parse가 없음** → **"1,5"→1.5 콤마-소수점/점-천단위 파싱을 직접 구현**해야
  함. 전 미터법 로케일에 영향, **DE에서 가장 첨예**. (로케일 인식 파서 필요.)

### 3.4 평 = 坪 = 동일 단위 `[3-0 확정]`
- 한국 평 · 일본 坪 · 대만 坪(ping) 모두 **정확히 400/121 m²**, 1평=1坪(비율 1). → **하나의 팩터로 KO·JA·TW
  전통 면적을 커버**, 로케일 간 공유 가능. **4자리 3.306 말고 분수 400/121 사용**(오차 ~0.007% 회피).

### 3.5 요리 컵 (선택 필수) `[3-0, 값은 권위 있음]`
- **US legal 240mL · US customary 236.588mL · metric(AU/NZ/CA) 250mL · Japan 200mL · UK ~284mL.**
  → 부피/요리 카테고리에서 **컵 기준을 선택**하게. 단일 "cup"은 모호·오류. (레시피 전체에 ~4% 누적.)

### 3.6 신발/의류 사이즈 — 보류 `[검증 통과 claim 0]`

## 4. UI/UX 패턴 `[3-0 확정 다수]`

- **2단 카테고리 구조로 클러터 회피:** unitconverters.net = "Express" 6개 코어(길이·온도·넓이·부피·무게·시간)
  전면 + "Full" 40+개를 그룹(Common/Engineering/Heat/Fluids/Light/Electricity/…)으로. → **작은 일상 세트 기본
  노출, 롱테일(공학·과학)은 "전체/고급"으로 접기.**
- **탭 개수 규칙:** Material 3–5개, Apple HIG ≤6개. 카테고리 12+개엔 **평탄한 탭바 불가** → 드롭다운/그리드/2단 필요.
- **from/to 듀얼 입력 + 실시간 타이핑**(§1). **swap · copy · share(퍼머링크)** 버튼.
- **멀티-타깃 패널**(카테고리 전 단위 동시 표시)로 "한눈에".
- **모바일:** 숫자 키패드(`inputmode="decimal"`), 접근성(라벨·aria).
- ⚠️ "탭+swap/copy/share+멀티타깃"을 하나의 "권장 패턴"으로 묶은 단일 소스 claim은 **반증**(0-3) → 위 권장은
  **개별 검증된 조각을 조립**한 것(하나의 권위 패턴이 아님).

## 5. SEO 아키텍처 — ⚠️ 미해결, 빌드에 큰 영향

- 제네릭 용어는 unitconverters.net·RapidTables·**Google 자체 인라인 위젯**이 지배 → 정면 승부 불가.
- **쐐기 = 현지화 포맷 + 전통/지역 단위**(KO 평·돈·근, JA 坪·畳). 경쟁사 미개척.
- **✅ 결정됨(2026-07-17):** **카테고리별 slug**(`length-converter`·`area-converter`·`weight-converter`…).
  하나의 공유 변환 엔진에 카테고리 프리셋을 주입하고, 각 slug가 전용 SEO 랜딩 — **지금 QR 그룹 패턴(primary
  카드 + 타입 네비, `group`)을 그대로 재사용.** 카테고리 검색 포착 + 배관 비용 적음. (쌍별 페이지 Omni 모델은
  v1엔 과임 → 보류.) 카테고리별 slug 안에서 URL 상태(`?from=&to=&v=`)로 쌍을 addressable하게 두면 롱테일도 일부 포착.
- **✅ 결정됨: 전통단위 전역 노출.** 평·돈·근·坪·畳 등을 **모든 로케일에 노출**(도구=전 로케일이라는 현재
  아키텍처와 바로 호환 — 로케일 스코핑 불필요). 전통단위는 어차피 미터법으로의 변환이라 무해.

## 6. 우리 프로젝트 함의 & 권장 v1

- **v1 범위:** 코어 8–9 카테고리(온도·길이·무게·부피·넓이·속도·시간·디지털저장). 요리 컵(기준 선택) 포함 추천.
- **전통단위:** 넓이에 평/坪(공유 팩터 400/121), 무게에 돈/匁·근(품목 선택)·貫·斤, 길이에 자·치/尺·寸. **KO/JA
  로케일에서 특히 가치** → 노출 스코핑은 §5 결정과 함께.
- **숫자 포맷:** 출력 `Intl.NumberFormat(locale)`, **입력은 로케일 인식 파서 직접 구현**(DE 콤마 우선).
- **구현:** 순수 JS 변환 로직(단위→기준단위 팩터 맵) + 단위 테스트(전통단위 팩터·근 다중값·콤마 파싱은 반드시
  테스트) → `ssr:false` client island → registry 엔트리 → 전 로케일 dict → per-tool SEO 랜딩. (extend-tools 규약)
- **범위 밖:** 환율/FX(백엔드). 신발/의류 사이즈(검증 데이터 확보 전 보류).
- **포지셔닝:** 회원가입 0·즉시·무료·오프라인(프라이버시 각은 변환기엔 약하나 유지).
