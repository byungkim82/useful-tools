# 이미지 압축기 — 구현 계획 (읽기용)

> 이 문서는 **처음 읽는 아키텍트·개발자**를 위한 깔끔한 요약본이다. 결정에 이르기까지의 리뷰·변동 이력은
> 생략했다. 근거 데이터는 `image-compressor-research.md`, 모든 결정과 그 이유·검증 로그는
> `image-compressor-implementation-plan.md`에 있다. 영어본: `image-compressor-plan-en.md`.

---

## 1. 한 문장 요약

100% 브라우저에서 도는 **배치 이미지 압축기**를 만든다. 파일을 드롭하면 브라우저 안에서 압축하고, 서버로 아무것도
업로드하지 않는다. 기존 도구 사이트(tools.solisapps.com, QR 생성기·단위 변환기)에 세 번째 스위트로 얹으며,
6개 로케일(한국어·English·Español·Português BR·日本語·Deutsch)로 낸다.

**왜 이길 수 있나:** 대형 브랜드(TinyPNG·iLoveIMG·Compressor.io 등)는 전부 파일을 서버로 업로드한다.
100% 클라이언트로 처리하면서 **광고 없고, 개수 제한 없고, 배치가 매끄러운** 조합은 아직 아무도 갖고 있지 않다.
이 빈틈이 우리 자리다. 프라이버시("파일이 기기를 안 떠남")는 특히 독일(DSGVO) 시장에서 강하게 먹힌다.

## 2. 무엇을 만드나

### v1 (첫 출시 — 날카로운 척추)
- **입력:** 드래그&드롭, 파일 선택, 작업 중 파일 추가
- **배치 큐:** 여러 장을 한 번에. 전역 설정(품질/포맷/리사이즈)이 큐 전체에 적용
- **압축 제어:** 프리셋 3종(고화질/균형/최소) + 품질 슬라이더
- **출력 포맷:** JPEG, WebP (둘 다 브라우저 Canvas가 네이티브 인코딩)
- **리사이즈:** 최대 치수 / 퍼센트 / 정확한 가로×세로 (비율 잠금)
- **메타데이터:** EXIF 자동 제거(기본 ON, 프라이버시+용량) + 사진 방향 자동 보정
- **결과:** 파일별 before/after 크기와 "-68%" 배지, 파일별 다운로드, **전체 ZIP 다운로드**
- **프라이버시 표면:** "업로드 없음·개수 제한 없음" 히어로 + "직접 검증하기"(와이파이 끄고도 작동, 네트워크 탭에
  이미지 전송 없음) + 무가입·무광고·무워터마크
- **접근성/테마:** 모바일 우선, 다크 모드, 기본 a11y

### v1.1 (곧바로 이어서)
이미지별 설정 오버라이드 · **타깃 파일 크기("정확히 100 KB 이하로")** · Ctrl+V 붙여넣기 · 폴더 업로드 ·
before/after 비교 슬라이더 · 유스케이스 프리셋(WhatsApp·이메일·증명사진 등, 타깃 크기에 의존해서 함께)

### v1.5 (이후)
**AVIF 출력**(더 강한 압축, WASM 지연 로드) · **PNG 출력 + oxipng 최적화**(아래 참고) · PWA/오프라인 설치

### 범위 밖
HEIC 입력(별도 도구로 분리 — 무거운 WASM 디코더 필요) · AI 배경 제거·업스케일·워터마크(각각 별개 제품) ·
실시간 환율 같은 서버 기능

> **PNG를 왜 v1에서 뺐나:** 브라우저 Canvas는 PNG를 oxipng 없이 재인코딩하면 크기를 못 줄이거나 오히려 키운다.
> 게다가 PNG는 무손실이라 품질 슬라이더가 아무 효과가 없다("고장난 것처럼" 보임). 그래서 **v1은 PNG를 입력으로만
> 받고 출력은 JPEG/WebP만** 제공한다. PNG 출력은 oxipng를 붙이는 v1.5에 복귀한다.

## 3. 어떻게 동작하나 (엔진)

모든 처리는 **브라우저 안에서** 일어난다. 서버 코드 0.

**핵심 원칙 두 가지:**
1. **Canvas 우선.** 압축·리사이즈·포맷 변환을 브라우저 기본 API로 처리한다:
   `createImageBitmap()`(디코드+방향 보정) → `OffscreenCanvas` → `convertToBlob()`(JPEG/WebP 인코딩).
   추가 라이브러리·다운로드가 필요 없어 초기 번들이 가볍다.
2. **무거운 건 나중에, 필요할 때만.** AVIF 같은 WASM 코덱(수 MB)은 사용자가 그 기능을 고를 때만 지연 로드한다.
   초기 로딩에는 절대 포함하지 않는다(성능 방어).

**스레드/성능:** 압축은 **Web Worker**에서 돌려 화면이 안 멈추게 한다. 이 방식이 static export 빌드에서 실제로
동작하는지는 **빌드 스파이크로 이미 검증했다** — `new Worker(new URL('./worker.ts', import.meta.url))`가 같은
도메인 청크로 정상 배포되고, 현재 CSP에서 그대로 로드된다. OffscreenCanvas를 지원 안 하는 구형 브라우저만
메인 스레드로 폴백한다.

**메모리 관리:** 12메가픽셀 사진 한 장이 디코드되면 약 48 MB를 차지한다. 그래서 이미지를 **한 장씩 또는 2~3장씩**
처리하고, 큰 이미지·모바일에서는 동시 처리를 1로 낮춰 iOS Safari의 메모리 킬(약 200~400 MB)을 피한다.
처리가 끝나면 즉시 메모리를 해제한다. 브라우저 Canvas 최대 크기(iOS 8192², 구형 4096²)를 넘는 사진은 자동으로
줄이되, **몰래 줄이지 않고 사용자에게 배지로 알린다.**

**타깃 크기(v1.1):** "이 이미지를 100 KB 이하로"는 품질 값을 이진 탐색으로 좁혀 맞춘다. 최저 품질로도 목표를
못 맞추면 치수를 줄여서 맞춘다.

## 4. 기존 코드베이스 위에서

이 도구는 **단위 변환기 스위트와 똑같은 구조**로 얹는다. 변환기가 8개 slug(길이·무게·온도…)가 하나의 공유
클라이언트를 쓰고 slug로 카테고리를 고르듯, 이미지 압축기도 여러 slug가 하나의 `ImageCompressorClient`를
공유하고 slug로 변형(어떤 포맷을 강조할지)을 고른다.

| 레이어 | 기존 패턴 재사용 |
|---|---|
| 도구 등록 | `src/tools/registry.ts`에 엔트리 추가 (`group: 'image'`, 아이콘, 키워드, 로더) |
| 클라이언트 마운트 | `ToolLoader`가 `next/dynamic({ ssr: false })`로 로드 — 브라우저 전용 코드가 정적 HTML에서 빠짐 |
| 도구 페이지 | 기존 `[slug]/page.tsx`가 SEO 메타·JSON-LD·정적 설명문을 처리. 이미지 그룹이면 `ImageTypeNav` 추가 |
| 다국어 | 도구 블록을 6개 로케일 사전에 추가 (구조 동일) |
| UI 문구 | 변환기의 `labels.ts`처럼 **코드에** 라벨 맵으로 보관. 로케일 JSON은 SEO 카피만 |
| 홈/네비 | 홈 그리드에 카드 1장, 도구 안에서 포맷 전환 네비 |

**반드시 지켜야 할 두 가지 규칙:**
1. 모든 도구 블록은 **6개 로케일에 구조가 완전히 동일**해야 한다(`check-i18n.mjs`가 강제). 타입 시스템이
   이걸 보장하는 기반이다.
2. 도구 DOM은 **`ssr:false`** — 정적 HTML에 들어가면 안 된다. Canvas/File API가 얹히는 자리다.

## 5. 파일 구성

`src/tools/image/` 아래에 둔다. **순수 로직(테스트 가능) ↔ 브라우저 전용 코드**의 경계를 명확히 나눈다.

```
src/tools/image/
  compress-math.ts       # 순수: 리사이즈 치수 계산, 바이트 포맷("1.2 MB"), 절감률, 출력 파일명, 프리셋 해석
  queue-reducer.ts       # 순수: 배치 큐 상태 전이 + 동시성 결정 (가장 위험한 로직 → 유닛테스트로 덮음)
  compress-math.test.ts  # 위 순수 함수들의 Vitest 유닛테스트
  encode.ts              # 브라우저: createImageBitmap → OffscreenCanvas → convertToBlob
  compress.worker.ts     # 브라우저: 워커 엔트리 (encode 사용, postMessage 프로토콜)
  useCompressQueue.ts    # 브라우저: 얇은 훅 — queue-reducer를 워커·타이머·메모리 해제에 배선만
  labels.ts              # UI 문구 라벨 맵 (로케일 오버라이드 포함)
  ImageCompressorClient.tsx  # island 루트 — 드롭존·설정·큐·결과를 조립
  ImageTypeNav.tsx       # 포맷 전환 네비
  components/            # DropZone, QueueItem, SettingsPanel, StatBadge …
```

> **테스트 전략:** 실제 인코딩은 브라우저 API라 유닛테스트가 안 되지만, **위험한 로직은 순수 함수로 빼서 테스트한다.**
> 특히 배치 큐(전이·동시성·취소·해제)와 리사이즈 계산은 가짜 인코더를 주입해 Vitest로 검증한다. 실제 인코딩·워커는
> 헤드리스 브라우저 검증(§8)으로 덮는다.

## 6. 인프라

- **CSP 변경은 딱 하나:** `public/_headers`의 `img-src`에 `blob:` 추가. 미리보기·썸네일·다운로드가
  `URL.createObjectURL()`(=`blob:` 스킴)을 쓰기 때문이다. **워커는 같은 도메인이라 현재 CSP로 이미 허용**되므로
  추가 설정이 필요 없다(스파이크로 확인). COEP는 켜지 않는다(분석 beacon 보호).
- **라이브러리:** `fflate`(약 8 KB, ZIP "전체 다운로드"용) 하나. WASM 코덱(AVIF 등)은 v1.5에 지연 로드.
  초기 island 번들 목표는 gzip 30 KB 이하.
- **배포:** `pnpm run deploy` (⚠️ `pnpm deploy`가 아니다 — 워크스페이스 빌트인이 스크립트를 가림).
  정적 export → Cloudflare Workers 정적 자산, 서버 비용 0.

## 7. 다국어 & SEO

- **slug 3개(v1):** `image-compressor`(대표) · `compress-jpg` · `compress-webp`. 셋 다 같은 클라이언트를
  공유한다. 도구가 17개 → 20개가 되고, 정적 라우트는 108 → **126**(6 로케일 × 21)로 는다.
- **로케일 = 번역이 아니라 시장별 각도.** 예: 독일은 "ohne Upload · DSGVO-konform", 브라질은 WhatsApp 첨부,
  한국은 "증명사진·이력서 사진 용량 줄이기", 일본은 "軽量化 · アップロードなし". 이 각도는 SEO 제목·설명문에만
  싣고, FAQ·특징 배열은 6로케일 구조를 동일하게 유지한다.
- **구조화 데이터:** 각 페이지에 `WebApplication` + `FAQPage` + **`HowTo`**(압축 단계). 도구 자체는 `ssr:false`라
  정적 HTML이 얇으니, 크롤 가능한 설명문(사용법·특징·FAQ)을 서버 렌더로 채워 검색 노출을 확보한다.

## 8. 검증 (짓고 나서 반드시 실행)

**빌드/구조:** `pnpm test && pnpm lint && pnpm build` + `check-i18n.mjs`(6로케일 구조 동일 → ALL GOOD).
빌드된 `out/`에서 3개 새 페이지의 제목·메타·JSON-LD, ssr:false 마커 부재, 홈 카드, 네비 링크, 사이트맵 URL 수 확인.

**실제 브라우저(헤드리스 + 배포 후 라이브):**
- 실제 압축 왕복: 이미지 드롭 → 출력이 원본보다 작음 → 포맷 변환(WebP) 동작
- WebP 인코딩 feature-detect(구형 Safari에서 조용히 PNG로 폴백하지 않고 JPEG로 명시 폴백)
- EXIF 제거 확인(출력에 GPS·방향 태그 없음) + 회전 사진이 안 눕는지
- 오버사이즈 사진이 크래시 없이 줄고 배지로 고지되는지
- 20장 배치가 메모리 초과 없이 완주하는지
- **프라이버시 실증:** 네트워크 탭에 이미지가 나가는 요청이 0인지
- CSP 위반 0, 분석 beacon 정상 로드, 6로케일 페이지 200

## 9. 실행 순서

1. **P0-b — 순수 코어 + 테스트.** `compress-math.ts`와 큐 리듀서 + Vitest. (브라우저 불필요, 여기부터 시작)
2. **P1 — 단일 이미지 island.** 드롭 1장 → 품질/포맷/리사이즈 → 다운로드. 워커·CSP 반영, 헤드리스 검증.
3. **P2 — 배치 + 편의(척추).** 큐·적응형 동시성·통계·ZIP·EXIF 토글.
4. **P3 — 등록/네비/홈.** registry 엔트리 3개, 포맷 네비, 홈 카드.
5. **P4 — 다국어 + SEO.** 한/영 원작성 → 나머지 4개 로케일 번역 → 구조 검사. HowTo 구조화 데이터.
6. **P5 — 검증 → 배포.** §8 전체 → `pnpm run deploy` → 라이브 재확인.

> **엔진 경로(워커 in static export)는 이미 스파이크로 확정**됐으므로, 남은 불확실성은 실제 기기에서의 메모리·품질
> 뿐이고 그건 P1~P2에서 실제로 돌려 보며 확인한다.

## 10. 남은 리스크

빌드·번들 계열 불확실성은 스파이크로 닫혔다. 착수 후 실제로 돌려 봐야 아는 것만 남았다:

- **모바일 메모리.** iOS Safari가 큰 사진·많은 배치에서 탭을 죽일 수 있다 → 적응형 동시성으로 방어하되 실기기 확인.
- **WebP Safari 꼬리.** 아주 구형 Safari가 WebP 인코딩을 조용히 PNG로 떨굴 수 있다 → feature-detect로 JPEG 폴백.
  확률은 낮지만 값싼 방어라 넣는다.
- **자동 방향 보정.** 브라우저마다 EXIF 방향 처리가 다르므로 실제로 회전 사진을 넣어 확인한다.

이 세 가지는 모두 "치명적이지만 확률 낮음"이 아니라 "확인만 하면 되는" 범주다. 방어책은 이미 설계에 들어있다.
